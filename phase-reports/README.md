## Workflow

### Standard Phase Completion
1. Phase agent finishes work and saves `PHASE_X_REPORT.md` in its own project root
2. User tells the orchestrator session (MBSPlatform) to review
3. Orchestrator fetches the report from the source project folder
4. Orchestrator reviews for anything that changes downstream phases
5. Orchestrator copies report here for record keeping
6. Orchestrator updates downstream instructions if needed, re-syncs to target folders
7. User proceeds to next phase only after green/yellow/red assessment

### Addendum Pattern (Post-Completion Work)
When a completed phase needs additional work:
1. Orchestrator writes new addendum items into the phase's `platform-instructions` doc
2. Orchestrator generates a paste-ready prompt for the agent
3. User runs the prompt in the project's Claude Code session
4. Agent generates `PHASE_X_ADDENDUM_REPORT.md` in its project root
5. Orchestrator reviews, copies here, cascades changes to downstream instructions
6. Naming convention: `PHASE_1_ADDENDUM_REPORT.md`, `PHASE_2_ADDENDUM_REPORT.md`, etc.

### Orchestrator Direct Changes
Sometimes the orchestrator session makes direct code changes to project folders (e.g., nav button additions, branding fixes). When this happens:
- Changes are committed and pushed from the orchestrator session
- An `ORCHESTRATOR_CHANGES.md` file is created in the project root documenting what changed and why
- The next agent session in that project should read `ORCHESTRATOR_CHANGES.md` to understand changes it didn't make

## How to Review a Report

When a report comes back, the orchestrator should check:
1. **Deviations from spec** — anything built differently than documented?
2. **New env vars** — do downstream phases need to know?
3. **API response shapes** — do they match what downstream consumers expect?
4. **Known gaps** — do any affect other phases?
5. **Gotchas** — does the report flag anything for downstream work?
6. **Field names** — do data model fields match the platform conventions?

After review, update:
- `SESSION_HANDOFF.md` — current state
- `CURRENT_STATUS.md` — known issues
- Downstream `platform-instructions-for-*/` files if anything changed
- Re-sync to target project folders: `cp -r platform-instructions-for-X/* Project/platform-instructions/`
